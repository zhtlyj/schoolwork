## 基于区块链的学生学业预警与学习干预系统（说明文档）

> 说明：本项目前后端分离，使用 **Next.js + MongoDB** 搭建后端 API，使用 **React（Vite）** 搭建前端界面，支持学生 / 教职工 / 管理员三种角色，并提供“手动下发预警 + 一键生成预警”的完整预警管理流程。

---

### 一、项目结构

- **backend**：Next.js App Router 后端服务  
  - `lib/mongodb.ts`：MongoDB 连接与错误处理  
  - `lib/jwt.ts`：JWT 签发与校验  
  - `models/User.ts`：用户模型（支持 `student` / `staff` / `admin`）  
  - `models/Warning.ts`：预警模型  
  - `models/Grade.ts`：成绩模型  
  - `models/Attendance.ts`：出勤模型  
  - `app/api/auth/*`：登录 / 注册等认证接口  
  - `app/api/students/*`：学生管理接口（CRUD）  
  - `app/api/warnings/*`：预警管理接口（列表、创建、删除）  
  - `app/api/warnings/candidates/route.ts`：**预警候选学生列表接口**（一键生成用）  
  - `app/api/grades/*`：成绩接口  
  - `app/api/attendance/*`：出勤接口  
  - `app/api/dev/seed-demo/route.ts`：**生成演示数据的 seed 接口**

- **frontend**：React 前端（Vite）  
  - `src/contexts/AuthContext.tsx`：登录状态与角色信息  
  - `src/pages/Home.tsx`：主页面（仪表盘 + 学生管理 + 预警管理等）  
  - `src/pages/Home.css`：首页及预警管理样式  
  - `src/pages/Register.tsx` / `src/pages/Login.tsx`：注册 / 登录页面  
  - `src/services/*.ts`：前端调用后端的 API 封装（`auth` / `students` / `warnings` / `grades` / `attendance`）

### 二、环境配置与启动方式

1. **环境变量（后端 `backend/.env.local`）**

```env
MONGODB_URI=mongodb://localhost:27017/schoolwork
JWT_SECRET=your-super-secret-jwt-key-change-in-production
JWT_EXPIRE=7d
```

2. **安装依赖**

```bash
# backend
cd backend
npm install

# frontend
cd ../frontend
npm install
```

3. **启动服务**

```bash
# 启动后端（默认 http://localhost:3000）
cd backend
npm run dev

# 启动前端（默认 http://localhost:5173）
cd ../frontend
npm run dev
```

4. **生成演示数据（成绩 + 出勤 + 学生）**

```bash
# 通过 curl 或 Postman 调用（需已登录 staff/admin，带上 Authorization 也可，根据实现）
curl -X POST http://localhost:3000/api/dev/seed-demo
```

seed 接口会生成：
- 若干学生（如“张三”“李四”）  
- 每个学生的多条成绩记录（包含低于阈值的成绩，用于触发预警）  
- 每个学生的多条出勤记录（包含缺勤次数超阈值的数据）

### 三、角色与权限设计

- **student（学生）**
  - 只能查看：自己的成绩、出勤、预警记录等
- **staff（教职工）**
  - 可管理学生：增删改查学生信息  
  - 可下发预警：对任意学生创建 / 删除预警  
  - 可使用“一键生成预警”功能  
- **admin（管理员）**
  - 拥有与 staff 相同的权限  
  - 额外具备系统设置、权限管理等高级配置入口（前端导航区分展示）  

登录后，前端根据 `user.role` 渲染不同导航菜单和页面能力。

### 四、预警管理模块（重点）

预警管理入口：前端 `Home.tsx` 中的“⚠️ 预警管理”，仅 **staff / admin** 可见。  
模块内分为两种模式：

#### 1. 手动下发模式（✍ 手动下发）

**目标**：由辅导员 / 教师手动选择学生和预警内容，下发定制化预警。

- 顶部模式切换区：
  - `✍ 手动下发`（当前模式）
  - `⚡ 一键生成`
  - “➕ 下发预警”按钮：打开预警创建弹窗

- **筛选区（默认展示全部预警，筛选是“过滤视图”）**
  - 学生：默认“全部学生”，可指定到某个学生  
  - 类型：默认“全部类型”（成绩 / 出勤 / 作业）  
  - 级别：默认“全部级别”（高危 / 中危 / 低危）  
  - 每页显示：10 / 20 / 50 / 100 条  
  - 当任意筛选条件生效时，会显示“🔄 重置筛选”按钮，一键恢复到“全部数据”视图。

- **预警列表**
  - 默认展示全部已下发的预警数据（按分页）  
  - 列包含：学生、类型、级别、课程、内容、创建者、时间、操作  
  - 类型 / 级别使用彩色标签区分，例如：
    - 类型：成绩 / 出勤 / 作业  
    - 级别：高危（红色）、中危（橙色）、低危（绿色）
  - 操作：删除（仅 staff/admin）

- **手动下发预警弹窗**
  - 字段：
    - 学生（下拉，默认空，必须选择）  
    - 类型（成绩 / 出勤 / 作业）  
    - 级别（高危 / 中危 / 低危）  
    - 课程  
    - 预警内容（textarea）  
  - 提交后：
    - 调用 `POST /api/warnings`  
    - 创建成功后刷新列表，展示“已成功下发预警”提示  

> 总结：**手动模式 = 默认展示全部预警数据 + 三维筛选（学生 / 类型 / 级别） + 手动新增 / 删除**。

#### 2. 一键生成模式（⚡ 一键生成）

**目标**：系统根据成绩 / 出勤等数据，自动筛选出“达到预警条件的学生名单”，提供操作按钮由老师决定是否下发。

- 顶部模式切换到 `⚡ 一键生成` 后：
  - 手动模式相关的预警列表隐藏  
  - 显示“预警候选学生列表”

- **一键生成的规则来源**
  - 从“系统设置 → 预警规则”中读取两个阈值（目前在前端以 `gradeThreshold` / `attendanceThreshold` 形式传入）：
    - 成绩预警阈值：例如 `< 60` 分 → 高危  
    - 出勤预警阈值：例如 `> 3` 次缺勤 → 中危  

- **后端候选学生接口**
  - 路由：`GET /api/warnings/candidates`
  - 核心逻辑：
    - 遍历所有 `Grade` / `Attendance` 记录  
    - 按“学生维度”聚合，生成结构：
      - `studentId` / `studentName` / 学号  
      - `gradeWarnings[]`：所有成绩不达标的科目  
      - `attendanceWarnings[]`：所有缺勤超标的科目  
    - 支持 `type` 查询参数（只看成绩 / 只看出勤 / 全部）

- **前端候选学生列表展示**
  - 默认：**所有达到预警条件的学生**  
  - 顶部筛选：
    - 预警类型：全部类型 / 成绩预警 / 出勤预警（只影响显示，不重新请求其他接口）  
  - 卡片结构：
    - 学生姓名 + 学号  
    - 成绩预警分组（📝 成绩预警（X 项））
      - 每条包含：课程名 + 学期 + 考试类型（期中 / 期末 / 平时）+ 分数  
      - 右侧“下发预警”按钮  
    - 出勤预警分组（📅 出勤预警（X 项））
      - 每条包含：课程名 + 缺勤次数 + 最近缺勤日期  
      - 右侧“下发预警”按钮  

- **一键下发流程**
  - 点击某条成绩预警的“下发预警”：
    - 前端调用 `POST /api/warnings`，参数自动根据记录生成：
      - type = `grade`，level = `high`  
      - course = 课程名  
      - message = 按模板拼出详细文案（分数 + 阈值）  
    - 下发成功后：
      - 刷新预警列表（供手动模式查看）  
      - 重新调用 `/api/warnings/candidates`，把已处理掉的项从候选列表中移除 / 更新
  - 出勤预警类似，type = `attendance`，level 通常设为 `medium`。

> 总结：**一键生成模式 = 默认展示所有“达到预警条件的学生名单”，老师在名单上针对具体的成绩 / 出勤项点击“下发预警”完成真正下发。**

### 五、学生管理与数据接入

- **学生管理**
  - 后端：`/api/students` 提供学生的增删改查  
  - 前端：`Home.tsx` 中的学生管理页支持搜索、分页、编辑、删除  

- **成绩 / 出勤数据**
  - 后端：
    - `/api/grades`：查询 / 新增 / 修改 / 删除成绩  
    - `/api/attendance`：查询 / 新增 / 修改 / 删除出勤  
  - 前端：
    - `gradeService.getGrades`、`attendanceService.getAttendance` 封装 API 调用  
  - 一键生成功能依赖这些真实数据；通过 `seed-demo` 可以快速生成一批用于演示和测试的数据。

### 六、系统设置（预警规则等）

在 `Home.tsx` 的“系统设置”页中，预留了多个配置分区：

- 预警规则：  
  - 成绩预警阈值  
  - 出勤预警阈值  
  - 未来可扩展作业、学习行为等规则  
- 干预方案库：  
  - 针对不同预警级别的干预策略（约谈、家长沟通、辅导课等）  
- 区块链配置：  
  - 区块链节点、合约地址等（目前以展示与占位为主，可用于论文描述）  
- 系统参数与权限：  
  - 可扩展角色权限、日志、审计等功能。

### 七、如何在答辩 / 论文中说明本系统

可以从以下几个角度进行展示和说明：

1. **功能流程**
   - 学生登录查看个人成绩 / 出勤 / 预警  
   - 教师登录进行学生管理、预警管理  
   - “手动下发”与“一键生成”两种预警发放方式  

2. **数据驱动的预警机制**
   - 使用成绩 / 出勤等客观数据 + 可配置规则  
   - 先生成“候选名单”，再由教师进行人工审核 & 下发  
   - 防止“一刀切”自动下发，体现“人机协同”特点  

3. **区块链设计思路（可在后续扩展实现）**
   - 预警记录（尤其是高危预警）上链，保证不可篡改  
   - 结合 `blockHash` 字段，与链上交易或事件进行关联  

4. **角色与权限**
   - 基于 JWT 的认证授权  
   - 前后端共同控制可见模块与可调用接口
